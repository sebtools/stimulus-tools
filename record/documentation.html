Record Controller

The Record Controller is a Stimulus controller for managing records via events for integration with data layers.

The Record Controller allows you to manage collections of record-like elements in the DOM. It tracks changes to fields marked with data-record-field, supports creating new records from templates, and broadcasts add/update/delete events to facilitate integration with external data layers. It can also update the DOM based on events it listens for that can be dispatched from your application code.

Record Hierarchy

The Record Controller can handle zero or more tables and records. So it is useful to explore the hierarchy of record elements.

  • Channel
    • Attribute indicators: data-record-channel
    • You can have multiple channels on a page, each managing its own set of records.
    • This provides a "channel" key in outgoing events.
    • Each controller uses only one channel but multiple controllers can listen to the same channel.
  • Table
    • Attribute indicator: data-record-table
    • Used to scope events and data operations.
    • This can be the same as the Controller element if only one table is in use.
    • This provides a "table" key in outgoing events. It may map to an actual table or be used in whatever way your receiving code requires.
  • Record
    • Attribute indicator: data-record-id
    • New records use an empty string data-record-id="" and will be assigned an ID when added.
    • This can be the same as the Table element if only one record is in use.
    • This provides an "id" key in outgoing events.
  • Field: An input or display element within a record, indicated by the data-record-field attribute. Represents a single field in the record.
    • Attribute indicator: data-record-field
    • Each element provides a key in the outgoing data with a name matching the data-record-field attribute value.
    • Values are read in this order (using the first available):
      • data-record-value attribute
      • data-value attribute
      • value property (for inputs)
      • textContent (for display elements)

Targets

Indicated by data-record-target attribute.

Target Element(s) Description
save button, input[type="button"], input[type="submit"]
 Optional save control target. Elements can be wired to record#save to perform an explicit save for a record.

Attributes

data- Attributes

All of these attributes are prefixed with "data-" in the attribute name.

Attribute Element(s) Default Value(s) Description
controller any record Attach the Record Controller to a container element.
action any record#save, record#delete, record#add Standard Stimulus actions that can be used to invoke controller methods from markup.

data-record- Attributes

All of these attributes are prefixed with "data-record-" in the attribute name.

Attribute Element(s) Default Value(s) Description
channel (container) (string) A logical channel name for the controller. Used in dispatched events to scope data operations.
table (container or record element) (string) A logical table/namespace name for the records. Used in dispatched events to scope data operations.
id (root record element) '' (string) The record identifier. New records use an empty string data-record-id="" and will be assigned an ID when added.
field any (string) The name of the field.
This will be the key in the collected record data object.
debounce (controller/container) 300 (integer ms) Milliseconds to debounce automatic saves when debouncing is used. Default is 300ms.
value field elements (string) The controller reads values in this order: data-record-value, data-value, the element's value property, then textContent.
ignore field elements true, in, out Controls whether a field is ignored for incoming updates (in), outgoing events (out), or completely (true).
auto-add (controller/container) false true If present (or set to "true"), the controller will ensure an empty record is maintained for adding new records.
template (controller/container) (selector) A selector to find a template element to clone when inserting new records. If not provided, the controller will look for a direct <template> child or an existing empty record.
add-position (controller/container) after before, after Where to insert new records relative to the empty record or existing records. Default is after.
defaults (record element) query-string Field defaults supplied as a query-string (for example: foo=bar&count=1). Parsed and applied to fields when creating new records.
names / ids (controller/container) pattern Patterns used to rewrite name and id attributes on inserted records. Patterns may include [name], [id], [record-id], and [num].
sort-detection (controller/container) event When set to event, the controller listens for a global sort event and will broadcast a record:ui:sort when detected.

Methods

The controller exposes several methods that can be called from application code. Commonly used methods are listed below.

Method Returns Arguments Description
save element (optional) Mark record fields as sent and dispatch an add or update event depending on whether the record has an id.
delete element (optional) Broadcasts delete event and removes the record element from the DOM.
insertRecordElement Element record (optional), id (optional) Insert a record element (from template or empty record) and optionally populate it with provided data and id.
load array, clearExisting (optional) Populate the controller from an array of {record, id} objects. Optionally clear existing records first.
getRecordData Object element (optional) Collects record and form values from a record element and returns { record, form }.
getRecordOrder Array element (optional) Returns an array of record ids in DOM order for the controller's table scope.

Filtering and Autoload

The controller supports declarative filters that are merged into load/ query requests. Filters are evaluated per-table and are used by the controller's load/query flow (see requestInitialLoad / getLoadFilter).

Where filters come from

  • Table-level static filter
    • Set on the table container via data-record-filter='{"key":"value"}.
    • Parsed as JSON and used as the base filter.
  • Dynamic filter elements
    • Any element inside the table scope with a data-record-filter attribute is considered a dynamic filter input.
      Note that while any element can be a filter, the data element would be the canonical choice for a filter element since it is not used for field inputs or display and is semantically appropriate for holding data.
    • Each filter element contributes a key/value pair to the filter object, where the key is the value of the data-record-filter attribute and the value is read from the element.
    • Key: the value of the data-record-filter attribute on the element.
      • data-record-idtable: If present, the controller finds the nearest ancestor record for the named table and uses that ancestor's data-record-id as the filter value. This is useful for master/detail flows.
      • data-record-urlparam: If present, the value is read from the URL query parameter with the given name instead of the element. This allows you to use a hidden element as a proxy for a URL-based filter.
      • data-record-filter-value: If present, this attribute's value is used directly as the filter value instead of reading from the element. This allows you to use an element as a proxy for a static filter value.
      • data-value: If present, this attribute's value is used as the filter value instead of the element's value or textContent. This allows you to use a display element as a proxy for a filter value.
      • Otherwise, the controller reads the element's value property (for inputs) or textContent (for display elements) as the filter value.

Top-level matching

The controller uses a "top-level" match algorithm (see getTopLevelMatches) when collecting filter elements so filters inside nested tables or nested scopes are ignored. This prevents accidental leakage of filter inputs from nested table components. Only the nearest matching filter elements under the table container are considered.

How filters are applied

  • Merging — the controller merges table-level JSON and dynamic filter inputs; dynamic inputs override keys from JSON.
  • Used for initial load — requestInitialLoad calls getLoadFilter and dispatches a record:ui:query (detail: { table, filter }). The data bridge or application should accept this filter and return matching records.
  • Bulk vs per-row — table-level filters are intended for bulk queries. Per-row loads (by id) are supported separately; they typically do not use the table-level filter object unless your bridge expects it.

Examples

Static JSON filter on the table:

<div data-controller="record" data-record-table="people" data-record-autoload="true"
		data-record-filter='{"active":true,"group":"admins"}'> ... </div>

Dynamic filters inside the table scope:

<input data-record-filter="status" value="active" />
<select data-record-filter="group">...</select>

Filter that uses an ancestor record id from another table:

<input data-record-filter="org_id" data-record-idtable="orgs" />

Notes:

  • Table-level data-record-filter must be valid JSON. Invalid JSON will be ignored with a console warning.
  • When a table-level autoload is enabled (for example data-record-autoload="true" on the controller or table), the merged filter object is dispatched with the load/query event so the data layer can return only matching records.
